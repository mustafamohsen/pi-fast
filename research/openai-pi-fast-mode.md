# Research: OpenAI Codex Fast mode + Pi extension APIs

## Summary
OpenAI’s current Codex docs define Fast mode as a Codex service-tier feature, not a Pi/OpenAI “thinking” level: supported Codex models can run ~1.5x faster at higher ChatGPT/Codex credit cost, toggled in Codex CLI with `/fast on|off|status` or persisted with `service_tier = "fast"` and `[features].fast_mode = true`. For a Pi extension, the relevant implementation surface is `pi.registerCommand("fast", ...)` for `/fast` UX plus `before_provider_request` to mutate the finalized provider payload before the HTTP request is sent; gate this carefully to OpenAI/Codex models that support the tier.

## Findings
1. **Fast mode semantics are service-tier based.** OpenAI says Fast mode increases supported model speed by 1.5x and consumes credits at higher rates; current support is GPT-5.5 and GPT-5.4, at 2.5x and 2x Standard credit rates respectively. It is separate from GPT-5.3-Codex-Spark, which is a distinct fast/less-capable model rather than a service tier. [OpenAI Codex Speed](https://developers.openai.com/codex/speed)
2. **Official Codex UX is `/fast on`, `/fast off`, `/fast status`.** OpenAI’s slash-command docs say `/fast` toggles a Fast service tier when the model catalog exposes one, reports current thread state, can be persisted when prompted, and is hidden when the current model does not advertise a Fast tier. [Codex slash commands](https://developers.openai.com/codex/cli/slash-commands)
3. **Persistent Codex config uses `service_tier` + feature gate.** OpenAI’s config reference lists `features.fast_mode` as enabling model-catalog service-tier selection in the TUI, including Fast commands, and lists `service_tier` as the preferred service tier for new turns. Built-in values include `flex` and `fast`; the reference also notes legacy `fast` config maps to request value `priority`, so implementers should verify the exact request value expected by Pi/OpenAI payloads before hard-coding. [Codex config reference](https://developers.openai.com/codex/config-reference)
4. **Do not claim API-key Fast-credit behavior.** OpenAI states Fast mode is available in Codex IDE, CLI, and app when signed in with ChatGPT; with an API key, Codex uses standard API pricing and cannot use Fast mode credits. A Pi extension should distinguish ChatGPT/Codex subscription auth from ordinary `OPENAI_API_KEY` usage if it exposes user-facing promises about cost/credits. [OpenAI Codex Speed](https://developers.openai.com/codex/speed)
5. **Pi extension commands are the right `/fast` entrypoint.** Pi extensions are TypeScript modules that can register custom commands with `pi.registerCommand("name", { description, handler })`; extension commands are checked before normal input processing. If multiple extensions register the same command, Pi keeps them with numeric suffixes. [Pi extensions docs](https://pi.dev/docs/latest/extensions)
6. **`before_provider_request` is the relevant mutation hook.** Pi fires `before_provider_request` after the provider-specific payload is built and just before sending; returning `undefined` leaves it unchanged, while returning a value replaces the payload for later handlers and the actual request. This is the safest documented place to add/remove a top-level `service_tier` field after confirming the active provider payload shape. [Pi extensions docs](https://pi.dev/docs/latest/extensions)
7. **Pi exposes current model/provider context.** Extension contexts include `ctx.modelRegistry / ctx.model`, and Pi has `model_select` events for model changes. Implementation can keep a status/footer indicator and re-check support when the user switches models. [Pi extensions docs](https://pi.dev/docs/latest/extensions)
8. **Pi provider/model docs matter for gating.** Pi has built-in subscription providers via `/login` including “ChatGPT Plus/Pro (Codex)” and API-key providers via env/auth file; OpenAI API key env is `OPENAI_API_KEY`. Custom models/providers can use `api: "openai-responses"` or `"openai-completions"`, but model config does not document a Fast/service-tier capability flag, so support detection likely needs a local allowlist and/or inspection of built-in model metadata at runtime. [Pi providers docs](https://pi.dev/docs/latest/providers), [Pi models docs](https://pi.dev/docs/latest/models)
9. **Packaging is straightforward if distributed.** A Pi package can declare resources in `package.json` under `pi.extensions`, or rely on conventional `extensions/`; Pi packages run with full system permissions, and Pi core packages such as `@earendil-works/pi-coding-agent`, `@earendil-works/pi-ai`, `@earendil-works/pi-tui`, and `typebox` should be peer dependencies with `"*"` if imported. [Pi packages docs](https://pi.dev/docs/latest/packages)

## Implementation implications
- Register `/fast` as an extension command accepting `on`, `off`, and `status`; reject/notify on unknown args.
- Store per-session or persistent state deliberately. For current-thread behavior, in-memory state plus session reconstruction may be enough; for persistence, use `pi.appendEntry()` or package/user config only if the UX explicitly asks for persistence.
- In `before_provider_request`, only add `service_tier` when Fast is enabled and the active provider/model is confirmed supported; remove/omit it when disabled.
- Start with an allowlist for known supported Codex models from official docs (`gpt-5.5`, `gpt-5.4`) and keep it easy to update. Avoid enabling for non-Codex, custom OpenAI-compatible, or API-key-only providers unless verified.
- Treat `service_tier = "fast"` vs request payload value (`fast` vs possible legacy `priority`) as a validation point before implementation; inspect Pi’s actual OpenAI/Codex payload with a dry-run/logging extension and no secrets.
- Do not equate Fast mode with Pi thinking/reasoning controls (`pi.getThinkingLevel()` / `setThinkingLevel()`); those are separate APIs.
